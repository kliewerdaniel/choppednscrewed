# Data Flow Diagrams

## Overview
This document contains textual representations of key data flows in the enhanced chatbot system. These diagrams illustrate how information moves between components during various operations.

## Core Data Flows

### 1. Basic Chat Interaction Flow
```
User → Frontend (Trippy-Chat) → Backend API → 
[Knowledge Base Query] → [Persona Activation] → 
Llama.cpp Inference → [Response Synthesis] → 
Backend API → Frontend → User
```

### Detailed Steps:
1. User types message in trippy-chat UI
2. Frontend sends message to `/chat/message` endpoint
3. Backend validates and stores user message
4. Backend queries knowledge base for relevant context (if enabled)
5. Backend activates relevant personas based on query analysis
6. Backend sends prompt to llama.cpp server with context and persona guidance
7. Llama.cpp generates response
8. Backend synthesizes response from multiple personas (if applicable)
9. Backend updates knowledge base with new information from interaction
10. Backend returns response to frontend
11. Frontend displays response to user

### 2. Knowledge Enhancement Flow
```
User Query → 
[Intent Analysis] → 
[KB Search: Entities/Relationships] → 
[Context Extraction] → 
[Prompt Enhancement] → 
Llama.cpp → 
[Response Generation] → 
[Information Extraction] → 
[KB Update]
```

### Detailed Steps:
1. User submits query
2. Backend analyzes intent to determine information needs
3. Backend searches knowledge base for relevant entities/relationships
4. Backend extracts key information from search results
5. Backend enhances prompt with retrieved context
6. Enhanced prompt sent to llama.cpp for generation
7. Response generated by LLM
8. Backend extracts new entities/relationships from response
9. Knowledge base updated with new information
10. Final response returned to user

### 3. Tool Execution Flow
```
Natural Language Request → 
[Intent Parsing] → 
[Tool Selection] → 
[Parameter Extraction] → 
[Safety Validation] → 
[Sandboxed Execution] → 
[Result Collection] → 
[User Feedback] → 
[Approval/Modification] → 
[Change Application] → 
[KB Update from Dev Insights]
```

### Detailed Steps:
1. User requests tool action via natural language in chat
2. Backend parses intent to determine desired action
3. Backend selects appropriate mistral-vibe tool
4. Backend extracts required parameters from conversation
5. Backend validates request against safety policies
6. Backend executes tool in isolated sandbox environment
7. Backend collects execution results, logs, and artifacts
8. Frontend displays results to user with option to approve/modify
9. User provides feedback or approves changes
10. Backend applies approved changes to codebase/files
11. Backend extracts development insights from changes
12. Knowledge base updated with new technical knowledge

### 4. Persona Management Flow
```
Interaction → 
[Performance Measurement] → 
[Score Calculation] → 
[Threshold Evaluation] → 
[State Transition] → 
[Metadata Update] → 
[Future Activation Adjustment]
```

### Detailed Steps:
1. User interacts with system (chat, tool use, etc.)
2. System measures performance metrics (relevance, satisfaction, task success)
3. Backend calculates composite performance score for active personas
4. Scores compared against promotion/demotion thresholds
5. Personas transition between states (active/stable/experimental/pruned)
6. Persona metadata updated with interaction results
7. Future activation decisions informed by performance history
8. Underperforming personas may be archived or modified
9. High-performing personas may be used as templates for new personas

### 5. Autonomous Learning Flow
```
Conversation → 
[Entity/Relation Extraction] → 
[Novelty Assessment] → 
[Source Verification] → 
[Knowledge Graph Integration] → 
[Connection Building] → 
[Confidence Scoring] → 
[Future Retrieval Availability]
```

### Detailed Steps:
1. Conversation occurs between user and system
2. Backend extracts entities and relationships from dialogue
3. New information checked against existing knowledge for novelty
4. Novel information verified against trusted sources (if enabled)
5. Validated new knowledge integrated into knowledge graph
6. System builds connections to existing related knowledge
7. Confidence scores assigned based on source reliability
8. New knowledge available for enhancing future responses
9. System may proactively query external sources for related information

## Data Transformation Examples

### User Input to Internal Representation
```
Raw Text: "How does photosynthesis work in plants?"
↓
Normalized: "how does photosynthesis work in plants?"
↓
Tokenized: ["how", "does", "photosynthesis", "work", "in", "plants", "?"]
↓
Lemmatized: ["how", "do", "photosynthesis", "work", "in", "plant", "?"]
⟶
Entities Extracted: 
  - photosynthesis (Concept, CONFIDENCE: 0.95)
  - plants (Entity Type: organism, CONFIDENCE: 0.87)
⟶
Relationships Extracted:
  - photosynthesis PROCESS_IN plants (CONFIDENCE: 0.82)
```

### Knowledge Graph to LLM Context
```
Knowledge Graph Subgraph:
  (photosynthesis:Concept) 
    -(PROCESS_IN)→ (plants:Organism)
    -(REQUIRES)→ (sunlight:Energy)
    -(PRODUCES)→ (oxygen:Molecule)
    -(PRODUCES)→ (glucose:Molecule)
    -(OCCURS_IN)→ (chloroplast:CellComponent)

↓
Context Text:
"Photosynthesis is a process that occurs in plants. 
During photosynthesis, plants convert sunlight into oxygen and glucose. 
This process takes place in the chloroplasts of plant cells."

↓
Enhanced Prompt:
"You are a helpful biology tutor. Use the following information to answer the user's question:
Photosynthesis is a process that occurs in plants. 
During photosynthesis, plants convert sunlight into oxygen and glucose. 
This process takes place in the chloroplasts of plant cells.

Question: How does photosynthesis work in plants?"
```

## Temporal Data Flow

### Message Lifecycle
```
t=0: User submits message
t=100ms: Frontend optimistically adds message to UI
t=200ms: Backend receives message via API
t=300ms: Message validated and stored in database
t=400ms: Knowledge base query initiated (if enabled)
t=500ms: Persona activation analysis completed
t=600ms: Context assembled from KB and conversation history
t=700ms: Prompt sent to llama.cpp server
t=800ms-2000ms: Llama.cpp generates response (streaming)
t=2100ms: First response chunk received by backend
t=2200ms-3500ms: Response chunks streamed to frontend via WebSocket
t=3600ms: Response generation completed
t=3700ms: Backend extracts entities/relationships from response
t=3800ms: Knowledge base updated with new information
t=3900ms: Final message metadata sent to frontend
t=4000ms: Frontend updates message with complete metadata
```

### Knowledge Update Propagation
```
t=0: New information added to knowledge base
t=100ms: Invalidation signal sent to query cache
t=200ms: Related embedding cache entries marked stale
t=300ms: Materialized views updated (if applicable)
t=400ms: Search index updated for full-text search
t=500ms: Notification sent to interested subscribers (WebSocket clients)
t=600ms: Analytics updated for knowledge growth metrics
t=700ms: Backup process queued for persistence
t=1000ms: Knowledge update fully propagated through system
```

These data flow diagrams illustrate the complex interactions between components in the enhanced chatbot system, showing how information moves from user input through processing and back to the user, while simultaneously updating the system's knowledge and capabilities.
